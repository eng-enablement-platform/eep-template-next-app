The full deployment stack is **Cloudflare** (domain + DNS) → **Vercel** (hosting) → **Clerk** (auth)

![Deployment diagram — Cloudflare DNS, Vercel, Clerk, and GitHub CI/CD](/code/deployment.png)

<Callout type='info'>
  CI/CD automation for database migrations is on the roadmap. For now, these
  steps cover a manual first deployment. Subsequent deploys happen automatically
  on every push to `main`.
</Callout>

<Callout type='warn'>
  **Clerk production instances require a custom domain you own.** `*.vercel.app`
  URLs are not supported — Clerk needs to set DNS records on your domain, which
  isn't possible on a domain Vercel owns. If you don't have a domain yet, skip
  to the [dev key quick start](#using-dev-keys-quick-start) below and come back
  once you've bought one.
</Callout>

---

## Using dev keys (quick start)

If you don't have a custom domain yet, you can deploy with Clerk **development** keys
(`pk_test_` / `sk_test_`) on the Vercel-generated URL. This gets the app running
immediately and is fine for personal testing. When you have a domain, create a
production Clerk instance and swap the keys out.

Limitations of dev keys:

- Stricter rate limits
- Email/SMS capped at 100 emails / 20 SMS per month
- Clerk shows a warning banner in the UI

## Step 3 — Deploy to Vercel

<CollapsibleSteps>

<CollapsibleStep title="Import your repository" defaultOpen={false}>

1. Go to [vercel.com/new](https://vercel.com/new). If this is your first time,
   Vercel will ask you to verify your email before continuing.

![Vercel email verification prompt](/vercel/vercel-verify-email.png)

2. Choose your Git provider. Select **GitHub** (or GitLab/Bitbucket if that's
   where your repo lives).

![Vercel Git provider selection](/vercel/vercel-gitlab.png)

3. Vercel needs the **GitHub App** installed to access your repositories. Click
   **Install** to begin.

![Vercel GitHub App install prompt](/vercel/vercel-github-install.png)

4. GitHub asks which account and repositories to grant access to. Install for
   the account that owns your repo — you can scope it to specific repos or all
   repos.

![GitHub App repository access screen](/vercel/vercel-install-app.png)

5. Vercel confirms the GitHub connection is established.

![Vercel GitHub connection verified](/vercel/vercel-github-verify.png)

6. You're now on the **Import Project** screen. Your connected repositories are
   listed here.

![Vercel Import Project screen](/vercel/vercel-import-project.png)

7. Find your repo and click **Import**. Vercel detects Next.js automatically
   and pre-fills the build settings — no changes needed.

![Vercel repo import selection](/vercel/vercel-import-repo.png)

<Callout>
  If you're deploying the docs site also, import the same repo a second time and
  set the **Root Directory** to `docs/`.
</Callout>

</CollapsibleStep>

<CollapsibleStep title="Set environment variables" defaultOpen={false}>

In the Vercel project → **Settings** → **Environment Variables**, add:

| Variable                            | Required                                        | Value                                   |
| ----------------------------------- | ----------------------------------------------- | --------------------------------------- |
| `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY` | Yes                                             | `pk_live_...` (production Clerk key)    |
| `CLERK_SECRET_KEY`                  | Yes                                             | `sk_live_...` (production Clerk secret) |
| `DATABASE_URL`                      | Yes                                             | Your Neon pooled connection string      |
| `NEXT_PUBLIC_DOCS_URL`              | No - Only needed if deploying docs site as well | `https://docs.yourproject.dev`          |

<Callout type='warn'>
  `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY` is inlined into the client bundle at build
  time — set it before the first deploy. Changing it later requires a rebuild.
  Use the production key (`pk_live_`), not the dev key (`pk_test_`). You'll get
  these from Clerk in Step 4 — you can set env vars after creating the project
  and before clicking Deploy.
</Callout>

</CollapsibleStep>

<CollapsibleStep title="Add your custom domain to Vercel" defaultOpen={false}>

Vercel Dashboard → your project → **Settings** → **Domains** → enter your domain
→ **Add**.

Vercel shows you the DNS records to add to Cloudflare:

| Record  | Name       | Value                  |
| ------- | ---------- | ---------------------- |
| `A`     | `@` (apex) | Vercel's IP address    |
| `CNAME` | `www`      | `cname.vercel-dns.com` |

Add these in Cloudflare: **DNS** → **Records** → **Add record**. Set both to
**DNS only** (grey cloud) — the orange Cloudflare proxy breaks Vercel's SSL
certificate provisioning.

DNS propagates in a few minutes. Vercel shows a green checkmark once it detects
the records and provisions SSL.

</CollapsibleStep>

<CollapsibleStep title="Add the docs subdomain (optional)" defaultOpen={false}>

If you're deploying the docs site as a separate Vercel project:

1. Docs Vercel project → **Settings** → **Domains** → add `docs.yourproject.dev`
2. Vercel gives you a `CNAME` pointing to `cname.vercel-dns.com`
3. Add that `CNAME` in Cloudflare DNS (`docs` name, **DNS only**)

</CollapsibleStep>

<CollapsibleStep title="Deploy" defaultOpen={false}>

Click **Deploy**. First build takes a minute or two. Once done your app is live.

</CollapsibleStep>

</CollapsibleSteps>

## Step 4 — Set up a production Clerk instance

Now that your domain is live, create the Clerk production instance pointed at it.

<CollapsibleSteps>

<CollapsibleStep title="Create a production application in Clerk" defaultOpen={false}>

Clerk Dashboard → **Create application** → choose your sign-in methods →
set the environment to **Production**.

![Clerk production instance setup](/clerk/clerk-setup-prod-instance.png)

Clerk offers the option to clone settings from your dev instance — this copies
your sign-in method configuration so you don't have to set it up twice.

![Clerk clone dev instance option](/clerk/clerk-clone-dev-instance.png)

When prompted for the **Application domain**, enter your custom domain (e.g.
`eep-next-template.dev`).

![Clerk application domain field](/clerk/clerk-app-domain.png)

Copy the production API keys — you can always retrieve these later from Clerk
Dashboard → **API Keys**:

- `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY` (starts with `pk_live_`)
- `CLERK_SECRET_KEY` (starts with `sk_live_`)

Add them to your Vercel project environment variables.

</CollapsibleStep>

<CollapsibleStep title="Configure the session token" defaultOpen={false}>

<Callout type='info'>
  If you cloned settings from your dev instance, this session claim is already
  configured and you can skip this step. You will still need to manually add the
  Admin role to your user in the production Clerk dashboard before you can
  access the full app — see [Granting Admin
  access](/docs/dev-environment/authentication#granting-admin-access) for the
  steps.
</Callout>

Do this once per Clerk instance — required for role-based access to work.

Clerk Dashboard → **Configure** → **Sessions** → **Customize session token** → Edit:

```json
{ "metadata": "{{user.public_metadata}}" }
```

</CollapsibleStep>

<CollapsibleStep title="Add Clerk DNS records in Cloudflare" defaultOpen={false}>

Once created, Clerk shows a setup checklist. The DNS setup is **only accessible
from this checklist** — there is no standalone Domains page in the sidebar.

![Clerk production setup checklist](/clerk/clerk-prod-setup-checklist.png)

<Callout type='warn'>
  If you close the checklist, reopen it via the **Tasks** icon in the top-right
  of the Clerk dashboard navbar.

![Clerk setup tasks icon](/clerk/clerk-setup-tasks.png)

</Callout>

<Callout>
  You don't need to complete the checklist immediately — close it and come back
  when ready.
</Callout>

Clerk provides 5 CNAME records for session management, email verification, and
auth flows. Copy each one from the checklist:

![Clerk domains setup](/clerk/clerk-domains.png)

<Callout type='idea'>
  **What is DNS and what is a CNAME?**

DNS (Domain Name System) is the internet's address book — it maps human-readable
names like `eep-next-template.dev` to the servers that handle requests for them.
A **CNAME** record says "this subdomain is an alias for another domain" — so
`accounts.eep-next-template.dev` pointing to `accounts.clerk.services` means
requests to your accounts subdomain get handled by Clerk's servers. This is how
Clerk runs auth flows that appear to live on your domain without you hosting
anything yourself.

</Callout>

Add each record in Cloudflare: **DNS** → **Records** → **Add record**. The
**Name** field is just the subdomain part (e.g. `accounts`, not
`accounts.eep-next-template.dev`) — Cloudflare appends the domain automatically.

You already have Vercel's records in this same panel — Cloudflare DNS holds all
records for your domain in one place regardless of which service they're for.
Vercel's records route traffic to your app; Clerk's records route auth subdomains
to Clerk's servers. They don't conflict.

<Callout type='warn'>
  Set every Clerk DNS record to **DNS only** (grey cloud, not orange) in
  Cloudflare. The orange proxy breaks Clerk's domain verification.
</Callout>

Once all records are added, hit **Check DNS records** in Clerk to verify.
Cloudflare propagates almost instantly.

</CollapsibleStep>

</CollapsibleSteps>

## Step 5 — Run production migrations

Vercel does not run migrations automatically. After the first deploy:

```bash
DATABASE_URL=postgresql://... pnpm db:migrate
```

<Callout type='info'>
  Automating migrations as part of the deployment pipeline is on the roadmap.
  For now, run them manually after each deploy that includes schema changes.
</Callout>

## Step 6 — Verify

Trigger a fresh Vercel deployment (push a commit, or redeploy from the dashboard)
to pick up the Clerk production env vars, then confirm:

- App loads at your domain
- Sign-in and sign-up flows complete without errors
- Google SSO redirects correctly (if enabled)

<Callout>
  If sign-in fails, the most likely causes are: Clerk DNS records not yet
  propagated, Cloudflare orange cloud proxy enabled on a Clerk record, or the
  GCP OAuth redirect URI not updated after adding your domain.
</Callout>

## DNS records summary

All records live in the same Cloudflare DNS panel:

| Type    | Name              | Value                         | Proxy    | Purpose             |
| ------- | ----------------- | ----------------------------- | -------- | ------------------- |
| `A`     | `@`               | Vercel IP                     | DNS only | App apex domain     |
| `CNAME` | `www`             | `cname.vercel-dns.com`        | DNS only | www redirect        |
| `CNAME` | `docs`            | `cname.vercel-dns.com`        | DNS only | Docs subdomain      |
| `CNAME` | `accounts`        | `accounts.clerk.services`     | DNS only | Clerk auth          |
| `CNAME` | `clerk`           | `frontend-api.clerk.services` | DNS only | Clerk Frontend API  |
| `CNAME` | `clk._domainkey`  | _(Clerk-provided)_            | DNS only | Email DKIM signing  |
| `CNAME` | `clk2._domainkey` | _(Clerk-provided)_            | DNS only | Email DKIM signing  |
| `CNAME` | `clkmail`         | _(Clerk-provided)_            | DNS only | Clerk email sending |

## Subsequent deploys

Every push to `main` triggers an automatic Vercel deployment. You only need to:

- Re-run `pnpm db:migrate` (against production `DATABASE_URL`) when a deploy
  includes new migrations
- Update Vercel env vars if new ones are added to the schema

## Troubleshooting

These are the issues you're most likely to hit, documented from going through
this setup end to end.

### `Failed to load Clerk JS` / `failed to load script: /__clerk/...`

The Clerk JS bundle loads via the `clerk` CNAME (`clerk.yourdomain.dev` →
`frontend-api.clerk.services`). This error means that record either hasn't
propagated yet or wasn't added.

- Confirm the `clerk` CNAME is in Cloudflare DNS set to **DNS only** (grey cloud)
- Even after Clerk shows DNS as verified, **redeploy on Vercel** — the running
  build needs to pick up the verified domain configuration
- Try a fresh incognito window after redeploying to avoid browser DNS cache

### `Unable to attribute this request to an instance` / publishable key error

This means the publishable key in your Vercel env vars either doesn't match
the production Clerk instance, or the instance hasn't fully activated yet.

- Confirm `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY` starts with `pk_live_` (not
  `pk_test_`) in Vercel → Settings → Environment Variables
- Make sure it's set for the **Production** environment, not just Development
- After updating the key, **redeploy** — this var is baked into the bundle at
  build time so updating it without redeploying has no effect
- If the key is correct and the error persists, **wait a few minutes** — Clerk's
  production instance takes time to fully activate after DNS verification.
  This is normal and will resolve on its own

### `www.yourdomain.dev` serving the app instead of redirecting

Vercel adds `www` as a separate domain by default. It needs to be set to
**redirect** to the apex domain, not serve the app independently. In Vercel
→ Settings → Domains, click `www.yourdomain.dev` and set it to redirect to
`yourdomain.dev` with a **308 Permanent Redirect**.

If Vercel blocks the redirect saying "another domain is already redirecting
here", the apex and www are configured in the wrong order. Remove the apex
domain, set up www as the redirect first, then re-add the apex as primary.

### Clerk DNS shows verified but the sign-in component still fails

Clerk's verification and the browser resolving the DNS are independent. Clerk
checks from its own servers; your browser checks from your local DNS resolver.
After Clerk verifies:

1. Flush your local DNS cache: `sudo dscacheutil -flushcache && sudo killall -HUP mDNSResponder`
2. Try an incognito window
3. Redeploy on Vercel to ensure the build reflects the verified state

### Build command / env vars

- Leave the Vercel build command as default — Vercel detects `pnpm-lock.yaml`
  and uses pnpm automatically
- `NEXT_PUBLIC_*` vars are inlined at **build time** not runtime — any change
  requires a full redeploy to take effect
- The `NEXT_PUBLIC_CLERK_SIGN_IN_URL` redirect vars are optional — they default
  to `/sign-in`, `/sign-up`, `/` via T3 Env and don't need to be set in Vercel
