The full deployment stack for this template is **Clerk** (auth) + **Neon** (database)

- **Vercel** (hosting) + **Cloudflare** (domain + DNS). All four have free tiers and
  wire together with minimal configuration.

<Callout type='info'>
  CI/CD automation for database migrations is on the roadmap. For now, the steps
  below cover a manual first deployment. Subsequent deploys happen automatically
  on every push to `main`.
</Callout>

---

## Before you start

- A [Vercel](https://vercel.com) account
- A [Clerk](https://clerk.com) account
- A [Neon](https://neon.tech) account with a project — see [Neon Setup](/docs/dev-environment/neon)

---

## Using dev keys (quick start)

You can get something running fast using Clerk **development** keys (`pk_test_` /
`sk_test_`). Clerk's dev instance works fine for early testing.

Limitations to be aware of:

- Stricter rate limits
- Email/SMS capped at 100 emails / 20 SMS per month
- Google SSO requires additional GCP setup (see below)
- Clerk shows a warning banner in the UI

When you're ready for real users, create a production instance and swap the keys
in Vercel — no code changes needed.

## Step 1 — Set up a production Clerk instance

<CollapsibleSteps>

<CollapsibleStep title="Create a production application in Clerk" defaultOpen={false}>

Clerk Dashboard → **Create application** → choose your sign-in methods →
set the environment to **Production**.

![Clerk production instance setup](/clerk/clerk-setup-prod-instance.png)

Clerk offers the option to clone settings from your dev instance — this copies
your sign-in method configuration across so you don't have to set it up twice.

![Clerk clone dev instance option](/clerk/clerk-clone-dev-instance.png)

When prompted for the **Application domain**, enter your custom domain (e.g.
`eep-next-template.dev`) if you have one, or your Vercel deployment URL
(`your-project.vercel.app`) if you're not using a custom domain yet. This can
be updated later but must match where your app is actually deployed.

![Clerk application domain field](/clerk/clerk-app-domain.png)

Once created, Clerk shows a setup checklist.

<Callout>
  You don't need to complete the checklist right now — you can close it and come
  back at any time from the Clerk dashboard. If you do work through it and hit
  the Google SSO step, follow the [GCP OAuth setup](#google-sso-optional) steps
  below for guidance on getting your Client ID and Secret.
</Callout>

![Clerk production setup checklist](/clerk/clerk-prod-setup-checklist.png)

</CollapsibleStep>

<CollapsibleStep title="Configure the session token" defaultOpen={false}>

Do this once per Clerk instance — it's required for role-based access to work.

Clerk Dashboard → **Configure** → **Sessions** → **Customize session token** → Edit:

```json
{ "metadata": "{{user.public_metadata}}" }
```

Save.

</CollapsibleStep>

<CollapsibleStep title="Set your production domain in Clerk" defaultOpen={false}>

Clerk Dashboard → **Configure** → **Domains** → add your Vercel deployment URL
(e.g. `your-app.vercel.app`) or your custom domain if you have one.

Clerk uses this to validate auth request origins — without it, sign-in will fail
in production. If you're adding a custom domain and need DNS verification, see
[Step 5 — Custom domain](#step-5--custom-domain-optional) for the Cloudflare DNS
records you'll need to add.

</CollapsibleStep>

</CollapsibleSteps>

### Google SSO (optional)

<Callout>
  Consult the [Clerk
  documentation](https://clerk.com/docs/guides/configure/auth-strategies/social-connections/google)
  if you require more information
</Callout>

<CollapsibleSteps>

<CollapsibleStep title="Create a GCP OAuth client" defaultOpen={false}>

1. Go to [console.cloud.google.com](https://console.cloud.google.com) → create or
   select a project
2. Navigate to **APIs & Services**

![GCP APIs and Services](/gcp/gcp-apis-services.png)

3. Go to **Credentials**

![GCP Credentials page](/gcp/gcp-creds.png)

4. **Create Credentials** → **OAuth client ID**

![GCP Create Credentials dropdown](/gcp/gcp-create-creds.png)

5. Application type: **Web application**. Under **Authorised redirect URIs**, add
   the Clerk callback URL — find the exact value in: Clerk Dashboard →
   **Configure** → **SSO Connections** → **Google** → **Redirect URI**

![GCP OAuth client creation form](/gcp/gcp-create-oauth-client.png)

6. Click **Create** → copy the **Client ID** and **Client Secret**

<Callout type='info'>
  On a Clerk **dev instance**, Google SSO works locally with zero GCP setup —
  Clerk uses shared credentials under the hood. If you do want to set this up
  for dev you will need a **separate GCP OAuth client** as dev and production
  have different redirect URIs.
</Callout>

</CollapsibleStep>

<CollapsibleStep title="Add the credentials to Clerk" defaultOpen={false}>

Clerk Dashboard → **Configure** → **SSO Connections** → **Google** → toggle on.

![Clerk SSO connections setup](/clerk/clerk-sso-setup.png)

Paste your **Client ID** and **Client Secret** → Save.

![Clerk GCP credentials entry](/clerk/clerk-gcp-details.png)

</CollapsibleStep>

</CollapsibleSteps>

## Step 2 — Create a Neon production database

Create a Neon project and get your pooled connection string. See

Use a **separate Neon project** for production — never point production at your
local or development database.

## Step 3 — Deploy to Vercel

<CollapsibleSteps>

<CollapsibleStep title="Import your repository" defaultOpen={false}>

[Vercel Dashboard](https://vercel.com/new) → **Import Git Repository** → select
your repo. Vercel detects Next.js automatically.

If you're deploying the docs site separately, import the same repo a second time
and set the **Root Directory** to `docs/`.

</CollapsibleStep>

<CollapsibleStep title="Set environment variables" defaultOpen={false}>

In the Vercel project → **Settings** → **Environment Variables**, add:

| Variable                              | Value                                   |
| ------------------------------------- | --------------------------------------- |
| `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY`   | `pk_live_...` (production Clerk key)    |
| `CLERK_SECRET_KEY`                    | `sk_live_...` (production Clerk secret) |
| `NEXT_PUBLIC_CLERK_SIGN_IN_URL`       | `/sign-in`                              |
| `NEXT_PUBLIC_CLERK_SIGN_UP_URL`       | `/sign-up`                              |
| `NEXT_PUBLIC_CLERK_AFTER_SIGN_IN_URL` | `/`                                     |
| `NEXT_PUBLIC_CLERK_AFTER_SIGN_UP_URL` | `/`                                     |
| `DATABASE_URL`                        | Your Neon pooled connection string      |

<Callout type='warn'>
  `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY` is inlined into the client bundle at build
  time — set it before the first deploy. Changing it later requires a rebuild.
  Use the production key (`pk_live_`), not the dev key (`pk_test_`).
</Callout>

</CollapsibleStep>

<CollapsibleStep title="Deploy" defaultOpen={false}>

Click **Deploy**. First build takes a minute or two. Once done, your app is live
at `https://your-project.vercel.app`.

</CollapsibleStep>

</CollapsibleSteps>

---

## Step 4 — Run production migrations

Vercel does not run migrations automatically. After the first deploy:

```bash
DATABASE_URL=postgresql://... pnpm db:migrate
```

<Callout type='info'>
  Automating migrations as part of the deployment pipeline is on the roadmap.
  For now, run them manually after each deploy that includes schema changes.
</Callout>

## Step 5 — Custom domain (optional)

<Callout>
  If you're happy with a `your-project.vercel.app` URL you can stop here —
  Vercel handles the domain for you and everything works out of the box. Come
  back to this step when you're ready for a real domain.
</Callout>

### Buy a domain and connect it to Vercel

[Cloudflare Registrar](https://www.cloudflare.com/products/registrar/) sells domains
at cost (no markup) and gives you free DNS management and SSL — the best value
registrar for this stack.

<CollapsibleSteps>

<CollapsibleStep title="Buy your domain on Cloudflare" defaultOpen={false}>

1. Go to [dash.cloudflare.com](https://dash.cloudflare.com) → **Domain Registration**
   → **Register Domains**
2. Search for your domain

![Cloudflare domain search results](/cloudflare/cloudflare-buy-domain.png)

3. Select your domain and proceed to purchase

![Cloudflare domain purchase form](/cloudflare/cloudflare-purchase-domain.png)

4. Complete the purchase — Cloudflare confirms and activates the domain

![Cloudflare domain confirmation](/cloudflare/cloudflare-domain-confirmation.png)

<Callout type='idea'>
  `.dev` is a natural fit for developer tools — Google-owned, HTTPS enforced by
  the registry (browsers won't load it without SSL), and signals the purpose
  immediately. `yourproject.dev` is a good pattern.
</Callout>

Once purchased, Cloudflare is automatically your DNS provider.

</CollapsibleStep>

</CollapsibleSteps>

### Connect your domain to Vercel

<CollapsibleSteps>

<CollapsibleStep title="Add the domain to your Vercel project" defaultOpen={false}>

Vercel Dashboard → your project → **Settings** → **Domains** → enter your domain
→ **Add**.

Vercel shows you the DNS records to add — copy them for the next step. For a
typical setup:

| Record  | Name       | Value                  |
| ------- | ---------- | ---------------------- |
| `A`     | `@` (apex) | Vercel's IP address    |
| `CNAME` | `www`      | `cname.vercel-dns.com` |

</CollapsibleStep>

<CollapsibleStep title="Add the DNS records in Cloudflare" defaultOpen={false}>

Cloudflare Dashboard → your domain → **DNS** → **Records** → **Add record**.

Add the `A` record and the `CNAME` from the previous step.

<Callout type='warn'>
  Set the proxy status to **DNS only** (grey cloud) on both records — the orange
  cloud (Cloudflare proxy) intercepts traffic before it reaches Vercel and
  breaks SSL certificate provisioning. You can re-enable it later once
  everything is working.
</Callout>

DNS propagates in a few minutes with Cloudflare. Vercel shows a green checkmark
in Domains once it detects the records and provisions the SSL certificate.

</CollapsibleStep>

<CollapsibleStep title="Add the docs subdomain (optional)" defaultOpen={false}>

If you're deploying the docs site as a separate Vercel project:

1. Docs Vercel project → **Settings** → **Domains** → add `docs.yourproject.dev`
2. Vercel gives you a `CNAME` pointing to `cname.vercel-dns.com`
3. Add that `CNAME` in Cloudflare DNS (`docs` name, **DNS only**)

</CollapsibleStep>

</CollapsibleSteps>

### Update Clerk for your custom domain

<CollapsibleSteps>

<CollapsibleStep title="Add your domain to the Clerk production instance" defaultOpen={false}>

Clerk Dashboard → **Configure** → **Domains** → **Add domain** → enter your domain.

Clerk asks you to add a DNS record to verify ownership. Add it in Cloudflare,
wait for verification, then set it as the primary domain.

</CollapsibleStep>

<CollapsibleStep title="Update the GCP OAuth redirect URI (Google SSO only)" defaultOpen={false}>

If you set up Google SSO, the GCP OAuth client has a redirect URI pointing to the
old Clerk domain. Add the new one:

1. [GCP Console](https://console.cloud.google.com) → **APIs & Services** →
   **Credentials** → your OAuth client
2. **Authorised redirect URIs** → add the new Clerk callback URL (find it in Clerk
   Dashboard → **Configure** → **SSO Connections** → **Google** → **Redirect URI**)
3. Save

</CollapsibleStep>

</CollapsibleSteps>

### DNS records summary

| Type            | Name               | Value                  | Proxy    | Purpose                   |
| --------------- | ------------------ | ---------------------- | -------- | ------------------------- |
| `A`             | `@`                | Vercel IP              | DNS only | Main app apex domain      |
| `CNAME`         | `www`              | `cname.vercel-dns.com` | DNS only | www redirect              |
| `CNAME`         | `docs`             | `cname.vercel-dns.com` | DNS only | Docs site                 |
| `TXT` / `CNAME` | _(Clerk-provided)_ | _(Clerk-provided)_     | DNS only | Clerk domain verification |

## Step 6 — Verify

Trigger a fresh Vercel deployment (or push a commit) to pick up any env var
changes, then confirm:

- App loads at your domain (or `your-project.vercel.app` if skipping custom domain)
- Sign-in and sign-up flows complete without errors
- Google SSO redirects correctly (if enabled)

<Callout>
  If sign-in fails after adding a custom domain, check Clerk Dashboard →
  **Configure** → **Domains** first, then the GCP OAuth client redirect URIs.
</Callout>

---

## Subsequent deploys

Every push to `main` triggers an automatic Vercel deployment. You only need to:

- Re-run `pnpm db:migrate` (against production `DATABASE_URL`) when a deploy
  includes new migrations
- Update Vercel env vars if new ones are added to the schema
